Here's a full breakdown of every parameter:

Allocation calculation (allocLookup → ALLOC table)
The table has 39 entries (index 0–38, representing 1–39 contracted hours/week). The row used depends on two flags:

Years of service (yearsService): 'lt5' (under 5 years) or 'gt5' (5 years or more). gt5 gives ~1h/week more across the board
Night shift eligibility (nightShift): 'no', 'yes', or 'auto'. 'yes'/'auto' uses the _night row which adds roughly 10% extra. 'auto' resolves to eligible only if the shift length ≥ avgShift
Contracted hours/week (contractedHours): the index into the row. 33h/week = index 32. Capped at 39h

The four rows are lt5_day, lt5_night, gt5_day, gt5_night.
Per-booking hours calculation (bookedHrs)
shift hours = end - start (handles overnight automatically)
break minutes = calcBreak(shift hours) — first matching break rule wins (sorted descending)
booked hours = round((shift hours - break minutes/60) × 4) / 4 — rounded to nearest 15 minutes
Default break rules: ≥9h→45min, ≥8h→30min, ≥6h→20min, ≥4.5h→15min. All configurable in Settings → Break Rules.
Financial Year structure

Each FY is exactly 364 days (52 weeks) from its start date (must be a Sunday)
fyRange(fyk) = startDate to startDate + 363 days
The anchor FY is FY2026/27 starting 2026-03-01. All other FYs are computed as multiples of 364 days from this anchor
Per-FY startDate overrides in fyAllocs take priority over the engine calculation — this is how you pin a specific Sunday if the 364-day engine drifts
Each FY also stores hours (the allocation) and carryOver (added to hours in getFYAlloc)
Total allocation = hours + carryOver

Periods (4-week blocks within an FY)

Period 1 starts on the FY start date
Each period = 4 weeks (28 days)
Maximum 13 periods per FY (13×4 = 52 weeks)
Used for calendar dividers, dashboard summary, and report grouping — not used in any allocation calculation

Stats / remaining hours (getStats)
remaining = allocation - taken - confirmed(future) - submitted(future) - planned
shifts left = floor(remaining / avgShift)

taken = confirmed bookings in the past
Refused and cancelled bookings are excluded from all totals
avgShift is used only to calculate "shifts left" — it has no effect on individual booking hours

Pro-rata (calcProRata)
pro-rata hours = round(full allocation × (days worked in FY / total FY days) × 2) / 2
Rounded to nearest 0.5h. Only affects the suggestion shown — you still manually enter the allocation.
Display/warning parameters

noticeDays (default 28): bookings within this many days show a ⚠ notice warning
yearEndWarnWeeks (default 8): FY end warning banner appears this many weeks before year-end
defaultStart/defaultEnd: pre-fill time selectors in the booking modal and Quick Setup

4-week rotation

4 weeks × 7 days, each day stores {on, start, end}
Week 1 = the week containing the current FY's start date
Used by isWorkDay() and getRotDay() to show the purple dot on calendar cells and to auto-fill booking times in "Full Shift (rota)" mode
Has no effect on allocation calculations — purely for convenience and the "not rota'd" warning

The answer is nuanced — there are actually two separate systems running in parallel:

System 1 — The 52-week engine (all FYs off the anchor)
getFinancialYearFromDate and getAutoFYStartDate both work by calculating FY_ANCHOR + (n × 364 days). Every FY's auto-computed start date is derived directly from the anchor — not chained off the previous FY. So:

FY2025/26 = anchor − 364 days
FY2026/27 = anchor (1 Mar 2026)
FY2027/28 = anchor + 364 days
FY2028/29 = anchor + 728 days

This means if the anchor drifts from your actual FY start (e.g. your real FY starts 2 Mar not 1 Mar), the error compounds — FY2028/29 would be wrong by the same 1 day, not progressively worse. They all trace back to the same fixed point.
System 2 — Manual start date overrides (each FY off the previous)
In renderFYAllocTable, when a new FY row is added and has no startDate yet, the suggested date is computed as:
jsnextFYStartDate(prevStart) // = prevFY.startDate + 364 days
This is chained — each new FY's suggested date is 364 days after the previous FY's stored startDate. And getFYStartDate always prefers the stored startDate over the engine calculation.
So once you've manually set or confirmed a start date for any FY, subsequent FYs chain off that stored date, not the anchor. This is actually correct behaviour — if your real FY started on 2 Mar 2026 (not 1 Mar), you correct it once and every future FY suggestion is also correct.
The practical consequence
If you've never touched the start dates in the FY table, everything runs off the anchor. If you've manually set even one FY's start date, that FY and all subsequent ones chain off the stored dates. The engine is only a fallback when no stored date exists.
This is actually a sensible design — the anchor gives you a working default, and the stored dates give you precision once you've confirmed them. The only risk is if the anchor is meaningfully wrong for your actual FY start, in which case any FY you haven't manually corrected will be slightly off.
